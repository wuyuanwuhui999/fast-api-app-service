# 项目背景
我有一个 FastAPI 多模块项目，结构如下：

## 模块列表
- common（共用模块，包含通用工具、配置、数据库连接等）
- user（用户模块）
- chat（聊天模块）
- prompt（提示词模块）
- tenant（租户模块）
- gateway（网关模块）

## 模块识别规则
URL 格式：`/service/{模块名}/...`
- `/service/user/getUserData` → user 模块
- `/service/prompt/getPrompt` → prompt 模块

## 模块内部结构
每个业务模块（user/chat/prompt/tenant/gateway）均包含：
模块名/
├── dependencies/ # 依赖注入、权限校验
├── models/ # 数据库模型（SQLAlchemy等）
├── repositories/ # 数据访问层（CRUD操作）
├── routers/ # 路由定义（API端点）
├── schemas/ # Pydantic 请求/响应模型
└── service/ # 业务逻辑层

## 架构特点
- 所有服务注册到 Nacos
- 网关模块负责路由转发和请求日志（异步写入数据库）

# 开发规范与约束（重要）

## 1. 统一接口返回规范
所有接口返回结果必须使用 `ResultUtil` 类进行包装，返回结果类从 `common` 模块导入。
- **导入路径**：`from common.utils.result_util import ResultEntity, ResultUtil`
- **成功返回**：使用 `ResultUtil.success` 方法包装。
- **方法参数**：
  - `data: Optional[Any] = None`：接口返回的实际业务数据。
  - `msg: Optional[str] = None`：提示信息（通常在错误提示，或增删改成功提示时使用）。
  - `total: Optional[int] = None`：记录总数（仅在分页查询接口时设置）。
  - `token: Optional[str] = None`：用户凭证（仅在登录、注册接口返回，其他接口无需返回）。

## 2. 服务注册规范
- 所有接口/服务都需要向 Nacos 注册服务。
- **服务名称命名规则**：`模块名称-service`（例如：user 模块的服务名为 `user-service`）。

## 3. 认证与身份透传规范（Gateway + 业务模块）
- **Gateway 模块**：
  - 作为统一入口，负责处理所有外部请求的 JWT Token 验证（从请求头 `Authorization` 中获取）。
  - 验证通过后，从 Token 中解析出 `userId`，并将其放入请求头（例如 `X-User-Id`）透传给下游业务模块。
  - **严禁** Gateway 直接调用业务模块的内部逻辑，只能通过路由转发。
- **业务模块（user/chat/prompt/tenant）**：
  - 业务模块**不需要**也不应该再次进行 JWT 校验。
  - 统一使用依赖注入获取当前登录人 ID。
  - **获取方式**：在路由函数参数中添加 `current_user_id: str = Depends(get_user_id_from_header)`。
  - `get_user_id_from_header` 的实现逻辑是从请求头中提取 Gateway 透传过来的 `userId`。

## 4. 代码结构与风格
- 代码需要遵循现有项目结构，文件放在正确的目录下。
- 遵循分层原则：router → service → repository → model。
- 使用异步语法（async/await）。
- 错误处理统一使用 `HTTPException`。
- **命名风格**：
  - 文件命名用 `snake_case`，命名规则为 `模块名称_分层名称.py`（例如：user路由命名为 `user_router.py`）。
  - 类名用 `PascalCase`。
